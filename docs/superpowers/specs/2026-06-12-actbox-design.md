# ActBox 设计文档

> 创建日期:2026-06-12
> 状态:已通过设计评审,待拆实施计划

## 1. 项目概述

**ActBox**(action + inbox,"行动箱")是一个**开源、面向个人、自托管**的工具,核心能力:

- 从**邮件**里自动抽取"需要你做的事",整理成一个待办清单
- 在**页面内闭环**管理待办(列表、截止提醒、勾选完成)
- 用 **AI 辅助回邮件**(起草 → 人工确认 → 发送)

**差异化定位**:中文场景特化(国产 LLM + 中文截止表达/委婉请求识别)、个人自托管、数据自主。

**未来愿景**(本次 MVP 不做,但架构预留):多源消息(飞书/钉钉)→ 待办聚合。消息源做成可插拔适配器,核心引擎与消息源解耦。

## 2. 核心约束(已确认)

| 维度 | 决定 |
|---|---|
| 目标用户 | 个人自用、开源 |
| MVP 范围 | 仅邮件(QQ/网易等,IMAP + 授权码);架构预留多源适配器 |
| 排除项 | 个人微信(无合规 API,红线,见 §11);飞书/钉钉留 v2 |
| LLM | 云端国产 API(DeepSeek / 通义千问,OpenAI 兼容接口) |
| 待办去向 | 页面内 todo 闭环(列表/提醒/完成) |
| 回邮件 | 阶段 5 实现;AI 起草 + **人工确认铁律**(绝不自动发) |
| 部署 | 单用户、Docker 自托管;不做多租户账号系统 |
| 技术栈 | Next.js 全栈(TypeScript),详见 §10 |
| 开发模式 | vibecoding(用户描述需求,AI 写代码),需专门护栏,见 §9 |

## 3. 整体架构

### 双向数据流

```
收件回路:  邮箱 ──IMAP──▶ 适配器 ──▶ 抽取引擎 ──▶ 存储 ──▶ 页面
回邮回路:  页面 ──▶ AI起草 ──▶ SMTP发送 ──▶ 邮箱
```

### 模块划分(5 个,各司其职,通过清晰接口通信)

| 模块 | 职责 | MVP |
|---|---|---|
| ① 适配器层 | 对接邮件源:**收(IMAP) + 发(SMTP)**,产出/消费统一消息格式 | ✅ 邮件;预留飞书/钉钉接口 |
| ② 抽取引擎 | 一段文本(邮件正文)→ 结构化待办;调 LLM,中文 prompt 调优 | ✅ 核心 |
| ③ 存储层 | SQLite 存待办 + 已处理邮件记录(幂等) + 配置 | ✅ |
| ④ 调度层 | 定时拉取邮件 + 手动"现在拉取";幂等去重 | ✅ |
| ⑤ Web 层 | 待办列表页、配置页;Next.js API Routes | ✅ |

## 4. 模块设计

### 4.1 适配器层(收 + 发)

- **统一消息格式**(RawMessage):`{ source, messageId, subject, from, body, receivedAt }`
- **收件**:`MailReceiver` 用 `imapflow` 连 IMAP,拉取未处理邮件,清洗正文(去 HTML/引用/签名),产出 RawMessage
- **发件**:`MailSender` 用 `nodemailer` 走 SMTP 发送(QQ/网易的 SMTP 与 IMAP 共用同一套授权码)
- **可插拔**:定义 `SourceAdapter` 接口,`MailAdapter` 实现;将来 `LarkAdapter`/`DingTalkAdapter` 各自实现,接入 ① 层即可,不动 ②③⑤
- **连接参数**:`{ host, port, user, authCode, tls }`,从配置/`.env` 读

### 4.2 抽取引擎(产品灵魂)

- **输入**:RawMessage 的正文(清洗后纯文本)
- **输出**:`Todo[]`,每条:`{ title, dueDate?, priority?, context? }`
- **实现**:后端调 DeepSeek/通义(OpenAI 兼容 SDK),**强制结构化输出(JSON)**,附带来源邮件引用
- **中文特化 prompt 要点**(差异化核心):
  - 识别中文截止表达:"下周五前""月底""尽快""节前"
  - 识别委婉请求:"麻烦帮看下""辛苦跟进"= 待办
  - **区分「只是告知」与「需要行动」**("已收到,谢谢"≠ 待办)
  - 多个待办分别成条,不合并

### 4.3 存储层

SQLite(单文件 `data.db`)+ Drizzle ORM,首次运行自动建表。详见 §5。

### 4.4 调度层

- `node-cron` 定时拉取(频率可配,默认如每 30 分钟)
- 手动触发:页面"📥拉取"按钮
- **幂等**:按邮件 `messageId` 去重,已处理的不再抽取(防重复,见 §11)

### 4.5 Web 层

- 主页:待办列表(分组/筛选/勾选完成/查看原文/回复),移动端友好
- 配置页:邮箱地址、IMAP/SMTP、LLM key、拉取频率
- Next.js App Router,API Routes 承载后端逻辑(拉取、增删改待办、起草/发送回复)

### 4.6 回邮件(阶段 5)

1. 在邮件/待办上点「回复」
2. AI 据原邮件 + 待办上下文,起草回复草稿
3. 用户过目、可改 → 点「确认发送」
4. SMTP 发出 + 留发送记录

> 🔒 **安全铁律:AI 只起草,永不自动发送。** 发邮件不可撤回,必须人工确认。此约束不可绕过。

## 5. 数据模型(SQLite)

| 表 | 字段(要点) | 用途 |
|---|---|---|
| `todos` | id, title, dueDate?, priority?, status(pending/done), sourceMessageId, sourceSubject, sourceFrom, createdAt, updatedAt | 待办主体 |
| `messages` | id, messageId(唯一,去重用), subject, from, receivedAt, processedAt, bodySnippet | 幂等去重 + 留痕 |
| `settings` | key, value | 邮箱地址、host/port、拉取频率等运行配置 |

> **密钥不入库**:邮箱授权码、LLM API key 放 `.env`,不进数据库、不进 git。

> 回邮件的发送记录:可在 `messages` 增加 `direction` 字段(in/out),或单独 `sent_messages` 表——具体在实施计划阶段定。

## 6. 分阶段实施路径(vibecoding 友好)

**原则:每阶段都能在浏览器看到效果 + 一个 git 提交;永远不让 AI 一次跨两个阶段。**

| 阶段 | 做什么 | 完成后可见 |
|---|---|---|
| **0 地基** | `create-next-app` 起项目、git 初始化、基础依赖 | 浏览器打开有空页面在跑 |
| **1 引擎** ⭐(先验证核心) | 文本框粘贴邮件 → 后端调 LLM 抽取 → 页面显示 | 粘贴邮件点按钮,列出抽出的待办 |
| **2 存起来** | 接 SQLite + Drizzle、待办存库、列表页、勾选完成 | 待办刷新不丢,可管理 |
| **3 接邮箱** | `MailReceiver`(imapflow)连 QQ/网易、"现在拉取"、幂等去重 | 点一下,邮箱待办自动进列表 |
| **4 自动化** | node-cron 定时拉取、截止提醒、配置页 | 挂着自动跑、到点提醒 |
| **5 回邮件** 🆕 | `MailSender`(nodemailer) + AI 起草草稿 + 确认发送 | 邮件上点回复→起草→确认→发出 |

阶段 5 内部小步:先"手动输入回复发送" → 再"AI 起草草稿" → 最后"快捷模板"。

## 8. 配置与认证

- **密钥**:`.env` 存 `LLM_API_KEY`、邮箱授权码、`ACCESS_PASSWORD`(可选)。`.env` 加入 `.gitignore`
- **访问保护**:单用户、无账号系统。localhost 可裸奔;部署到服务器时用 `ACCESS_PASSWORD` 挡一层

## 9. 错误处理与测试(vibecoder 护栏)

**错误处理**(所有错误进日志,方便贴给 AI 看):
- IMAP 连不上/密码错 → 页面提示「连接失败」,不崩
- LLM 超时/失败 → 自动重试,该封先跳过
- 模型未按格式吐 JSON → 容错解析,记日志

**测试**(不追求全覆盖,锁死核心):
- **金标准测试**:5~10 封典型中文邮件 + 期望抽取结果,每次改 prompt 跑一遍,确保没把抽对的改坏
- 其余靠每阶段在页面上"眼见为实"

**vibecoding 三护栏**(贯穿全程):
1. 每完成一小步即 git 提交(生命线,改崩能退)
2. 小步迭代,每功能先能跑能看到再加下一个
3. 学会看运行日志/报错,出问题贴给 AI 读

## 11. 关键设计决策与理由

1. **适配器模式预留多源**——飞书/钉钉将来各做一个 `SourceAdapter`,接 ① 层即可,核心 ②③⑤ 不动。这是"多源愿景"能落地的唯一可持续架构。
2. **幂等防重复**——按邮件 `messageId` 去重。否则定时拉取会把同一封邮件的待办重复抽取(邮件类工具最常见坑)。
3. **回邮件人工确认铁律**——发邮件不可撤回,AI 只起草,必须人工点确认。
4. **先做引擎再做邮箱**(阶段 1 先于 3)——抽取准确率是核心价值,最早用"手动粘贴"验证,不准就早调 prompt,不必等整套做完。
5. **SQLite 而非 Postgres**——零安装(就一个文件)、自动建表、备份即复制文件;个人自用绰绰有余,真到多人再换。
6. **个人微信排除**——腾讯无官方 API,抓取方案均违反协议、封号高风险、法律灰色,作为开源产品不可行。折中路线(企业微信 / 转发给机器人)留待将来评估。

## 12. 非目标(YAGNI)

本次**不做**:
- 多租户账号系统(个人单用户)
- 个人微信接入
- 飞书/钉钉消息接入(v2)
- 待办同步到外部工具(滴答/Notion 等)——页面内闭环已满足
- 复杂权限/团队协作
